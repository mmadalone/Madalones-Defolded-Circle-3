# Codebase Audit вЂ” v1.4.10 baseline

**Date:** 2026-04-27
**Mode:** AUDIT (per STYLE_GUIDE В§2.3)
**Scope:** `main` branch HEAD `ebba5ac`, post-v1.4.10 deploy. Excludes `feature/avatar`, LCARS theme repo, `test/matrixrain_preview/` Docker preview.
**Brutality:** No glazing (CLAUDE.md В§AUDITING).

## Executive summary

**Overall: B+** вЂ” a well-disciplined fork with unusually mature documentation hygiene for a single-developer project. The screensaver mod is professionally architected (single GPU draw call, atlas caching, 108-property config singleton via macro pattern, `displayOff` gating). Custom-vs-upstream is precisely tracked in `docs/CUSTOM_FILES.md`. Test infrastructure exists and is non-trivial. CI is wired (`build.yml`, `test.yml`, `tidy.yml`, `code_guidelines.yml`).

Where it loses marks: **complexity-budget violations are systemic** (`matrixrain.cpp` is 3Г— over the documented 500-line ceiling at 1,428 lines; 8 custom QML files exceed the 300-line ceiling). **Upstream-inherited security and performance practices are sloppy** (`VerifyNone` on TLS, QSettings file I/O on every property read in hot paths). And **the orphan-signal class of bug we just shipped a fix for in v1.4.10 is unlikely to be the only one** вЂ” `core::Api` declares 30+ signals and only `entityAdded` was systematically verified.

Per-dimension grades follow.

## В§1 Architecture / layering вЂ” Aв€’

**Strengths**
- Clean C++/QML split per STYLE_GUIDE В§1.2. C++ owns rendering, simulation, config bridges; QML owns visual composition. No simulation logic in QML, no UI layout in C++.
- Mod independence is real: Mod 1 (screensaver) and Mod 3 (battery chip) share zero code. Removing either is mechanical.
- Config bridge singleton pattern (`ScreensaverConfig`) cleanly separates upstream `Config` from custom config.
- Singleton boundaries are proper (`Q_ASSERT(s_instance == nullptr)` + `qmlRegisterSingletonType`).

**Weaknesses**
- вљ пёЏ **`matrixrain.cpp` at 1,428 lines violates В§1.6 complexity budget by 3Г—.** The file holds the renderer, the QSG material, the QSG shader, the QSG node subclass, the simulation orchestrator, the chaos burst handlers, the enter-button state machine, and the ScreensaverConfig binding вЂ” at minimum the QSG material/shader/node classes (`MatrixRainMaterial`, `MatrixRainShader`, `MatrixRainNode`, lines 53-142) belong in their own file. This was extracted partially (`layerpipeline.cpp`, `atlasbuilder.cpp`) but the main file is still a multi-responsibility monster.
- вљ пёЏ **`entityController.cpp` at 745 lines** вЂ” most of that is upstream constructor boilerplate (lines 22-161 are 140 lines of `qRegisterMetaType`/`qmlRegisterUncreatableType` calls), but it's pristine upstream code we shouldn't touch. Note for the record only.

## В§2 Correctness / bug surface вЂ” B

**Strengths**
- Every visual `QQuickItem` subclass sets `ItemHasContents` (verified: only `MatrixRainItem` is visual, `setFlag` at `matrixrain.cpp:147`). `InputController` is the other QQuickItem subclass but is non-visual.
- `MatrixRainNode` destructor correctly cleans up the atlas texture on the render thread (`matrixrain.cpp:135-141`). Comment explicitly notes the OwnsMaterial sequencing вЂ” defensible code.
- `Q_OBJECT` macro present on all custom QObject subclasses (`ScreensaverConfig`, `GravityDirection`, `MatrixRainItem`).
- v1.4.10's `entityAdded`/`onEntityChanged` fix is correct and well-documented.
- **Systematic `core::Api` orphan-signal audit (executed 2026-04-27 post-v1.4.10):** of ~50 declared event signals, only **2 genuine orphans** remain вЂ” both upstream-incomplete stub features (`voiceAssistantsChanged` at `core.cpp:2532`, part of an unfinished voice-assistants-list-management path where `getVoiceAssistants()` is also never called and `respVoiceAssistants` is declared-but-never-emitted; `respDockSetupProcesses` at `core.cpp:3038`, response to a `get_dock_setup_processes` request that upstream itself comments `TODO(marton): Implement me` at `enums.h:164`). All 12 `cfg*Changed` signals + the 9 cfg-slots in `Config` are correctly wired at `config.cpp:19-30`. The v1.4.10 fix was the only fork-relevant orphan. Class is closed.

**Weaknesses**
- вљ пёЏ **`EntityController::onCoreConnected/Disconnected` calls `m_entities.clear()` without disconnecting per-entity signals or deleting the QObjects** (`entityController.cpp:318-330`). Entities are children of `EntityController` so they survive the clear (no use-after-free), but they remain alive forever as long as the controller does, with stale signal/slot wiring against an entity-id that's no longer in the map. Not crash-class вЂ” but every reconnect leaks N entity objects + their slot connections. Over a long uptime with flaky network this accumulates.
- вљ пёЏ **`setPreviewImage` placeholder substring matching is Kodi-specific** (`mediaPlayer.cpp:499-516` вЂ” 12 hard-coded `Default*.png` filenames). If Plex/Spotify/etc. push similar placeholder URLs, the guard won't fire. The class is named `isKodiDefaultPlaceholder` honestly, so the scope is documented вЂ” but in practice this is a known gap.
- вљ пёЏ **`screensaverconfig.cpp:50` Battery deferred-connect uses `QTimer::singleShot(500, ...)`** to wait for `Battery` singleton to come up. 500ms is a magic number. If the battery comes up later than that, `showBatteryChanged` never fires post-Battery-init for clients that bound before. Minor вЂ” but the fix is to subscribe to a more reliable lifecycle signal.
- в„№пёЏ **`m_timer.start()` is called from at least 7 places in `matrixrain.cpp`** (lines 565, 1024, 1047, 1163, 1171, 1189, 1316) but only `setDisplayOff` (line 1336) and one branch (`if (m_running)` at 1047 / 1189) gate on `displayOff`. The other call sites assume the timer was either already running or will be paused later. This is fragile вЂ” a chaos-burst trigger arriving while `displayOff` is true could restart the timer. **Recommend:** consolidate timer start into a single `startTimer()` helper that always checks `m_displayOff && !m_running` first.

## В§3 Performance вЂ” B

**Strengths**
- Single-draw-call architecture in matrixrain (`updatePaintNode` builds one `QSGGeometryNode` with one material). Atlas reuse via `AtlasBuilder` static cache (`atlasbuilder.cpp`).
- Per-cell brightness map precomputed (per the screensaver implementation doc).
- `displayOff` gating present on the only renderer (`matrixrain.cpp:1336`).
- Vertex buffer is reused in-place вЂ” only UVs updated per frame, not full geometry rebuild (per design doc).

**Weaknesses**
- вљ пёЏ **`ScreensaverConfig` getters read QSettings on every call.** Every `showBattery()`, `speed()`, `density()`, `fadeRate()`, `trailLength()`, `color()` access (and all 108 SCRN_* macro getters) hit the QSettings file. Per `.claude-memory/feedback_uc3_disk_io.md`: "UC3 disk I/O is slow" вЂ” and these are called from QML bindings that re-evaluate on signal changes. On hot paths (per-frame property reads from `MatrixRainItem` bound to `ScreensaverConfig.speed`) this is a real cost. **Recommend:** cache the int/string in a member variable, refresh on setter, return cached value on getter. ~30 lines of edit; preserves the dual-emit pattern.
- вљ пёЏ **`getMediaImageColor` calls `m_nam.clearAccessCache()` and `clearConnectionCache()` on every fetch** (`mediaPlayer.cpp:460-461`). These clear connection pools вЂ” defeating HTTP keep-alive entirely. Every artwork fetch incurs a fresh TLS handshake. On a Kodi browse with rapid item switching this is meaningful latency. Inherited from upstream UC, but worth flagging вЂ” fixing it would mean hand-merging on next upstream pull.
- в„№пёЏ Toolchain image `unfoldedcircle/r2-toolchain-qt-5.15.8-static:latest` is unpinned. If UC bumps it, builds may differ silently. Already flagged in `_build_logs/2026-04-27_v1.4.10.md`.

## В§4 Security вЂ” C+

**Strengths**
- Credential handling clean: `.env.local` gitignored (verified at `.gitignore:111`), `${UC3_PIN}` interpolation throughout BUILD.md / scripts. No hardcoded PINs.
- HTTP basic auth on canary deploy is fine for a LAN device; RELEASE_SIGNING.md exists for binary verification.
- Logdy disabled by default (`autostart:false`); enabled explicitly when needed and disabled at end of session.

**Weaknesses**
- вљ пёЏ **`setPeerVerifyMode(QSslSocket::VerifyNone)` on TWO TLS paths**:
  - `voice.cpp:90` вЂ” voice cloud requests
  - `mediaPlayer.cpp:470` вЂ” media artwork fetches

  This silently accepts any certificate, including MITM. For artwork it's fine (image bytes, low risk). For the voice cloud it's worse (potentially STT/TTS audio + auth tokens). **Both inherited from upstream UC.** Per CLAUDE.md В§10, fixing means upstream drift on critical files вЂ” but this is genuinely a security deficiency, not a stylistic one. Worth raising upstream as an issue (declined per "keep mine-only" decision, but flagging anyway).
- вљ пёЏ **No input validation on incoming WS messages.** `core::Api::processEntityChange` blindly trusts attribute types from the wire (`core.cpp:2100-2163`). A malformed/malicious integration could crash the firmware via type confusion (e.g., passing a list where a string is expected). The integration is on the same LAN behind the user's auth вЂ” low practical exposure вЂ” but defense-in-depth would be a guard layer. Inherited from upstream.
- в„№пёЏ **`logs/mcp_usage.jsonl`** that we just gitignored contains tool-call telemetry from prior Claude Code sessions вЂ” historically had query parameters that could include sensitive data. Not exposed publicly (private repo) but worth noting.

## Top 5 actionable fixes (ranked by ROI)

1. ~~**вќЊ Systematic orphan-signal audit on `core::Api`.**~~ **Done same day.** Of ~50 event signals only 2 remain orphan and both are upstream-stub features (see В§2). Class is closed; no fork-side action.

2. ~~**вљ пёЏ Cache `ScreensaverConfig` raw values**~~ **Re-evaluated 2026-04-27, downgraded to в„№пёЏ.** Original framing conflated `QSettings::value()` with disk I/O вЂ” Qt 5.15 `QSettings` reads from an in-memory `QHash` (file load is lazy + mtime-refresh, rare in single-writer scenarios). Per-getter cost is ~1-3Вµs (QHash lookup + QVariant boxing), not full file I/O. Hot path = ~100 reads/sec в†’ ~300Вµs/sec в‰€ 0.03% CPU saved. Not worth the ~80-line diff that breaks SCRN_* macro uniformity for 7 of 108 properties.

3. **вљ пёЏ Consolidate `m_timer.start()` calls in `matrixrain.cpp` into a single helper that gates on `displayOff`**. 7 callsites currently; if any of them fires while `displayOff` is true, the AP-UC-08 В§7.5 contract is violated. ~20 lines. Defensive; prevents a class of regressions when adding new effects.

4. **вљ пёЏ `EntityController::onCoreDisconnected/Connected` should `deleteLater()` entity objects, not just `m_entities.clear()`.** Stale entity QObjects accumulate over flaky-network sessions. ~15 lines: iterate map, deleteLater, then clear. Defensive; eliminates a slow-leak class.

5. **в„№пёЏ Pin the toolchain image** to a specific digest or fork-managed tag. Two-line edit to `BUILD.md` + `_build_logs/` template. Locks reproducibility.

Honorable mentions (not in top 5 but worth queueing):
- Extract `MatrixRainMaterial`/`Shader`/`Node` into `matrixrain/material.{h,cpp}` (-90 lines from main file, +1 modular unit).
- Re-run `docs/A11Y_AUDIT.md` against post-v1.4.0 BaseDetail layout.

- Generalize `isKodiDefaultPlaceholder` if a non-Kodi integration ever hits the same pattern вЂ” currently scoped to Kodi by name, so honest.

## Things deliberately NOT to change

- **`setPeerVerifyMode(VerifyNone)` on artwork + voice TLS paths.** Inherited from upstream UC. Fixing means upstream drift on `mediaPlayer.cpp` and `voice.cpp` (one of which we already drift heavily). Real security weakness, but fork-keep-mine policy + drift-minimization makes it a no-fix. Issue documented here for awareness.
- **Hand-written setters in `screensaverconfig.cpp:107-140`** for the 5 transformed-property backers. Tempting to "clean up" with macros. The Qt 5.15 MOC quirk that motivated them is real and documented. Don't.
- **`m_entities.clear()` on `onCoreConnected` / `onCoreDisconnected`** as written. Upstream pattern. The fix would expand to deleteLater + clear вЂ” but that's a behavior change, not a cleanup; treat as a separate decision.
- **`SCREENSAVER-IMPLEMENTATION.md` length.** It's 2,290 lines because the screensaver IS that complex. Splitting it just for size would degrade discoverability.
- **Translation `.ts` regen on every release.** It's noisy in diff, but lupdate must run for new qsTr() strings to land in non-English locales. Suppressing the regen would silently break i18n.
- **Per-property NOTIFY signal indirection on the 5 dual-emitters.** The Q_PROPERTY-named signal is the one QML bindings follow. Indirect chains via `connect(this, &X::aChanged, this, &X::bChanged)` looked cleaner pre-2026-04-14 but didn't actually wire through QML. Don't reintroduce.
