## v1.4.7 — TouchSlider screensaver guard completeness

### Origin

User-reported mid-session, post-v1.4.6 deploy: "while the screensaver is running the slider is very sensitive. can we tune it down?" — after first noting that *"when the screensaver is running and the slider is used, the screensaver speed is changed but the slider has effect on kodi. this is not desired."*

### Root cause

All 4 `TouchSlider*.qml` variants (Volume, Seek, Brightness, Position) had the `applicationWindow.screensaverActive` early-return guard **only on `onTouchPressed`**. When the screensaver owned the touchbar:

- `onTouchPressed` returned early — `prevTouchX` and `targetVolume` never initialized to fresh values (good so far).
- `onTouchXChanged` ran ungated — accumulated `targetVolume += Math.sign(rawDelta)` against stale state.
- `onTouchReleased` ran ungated — committed the stale target via `entityObj.setVolume(sliderContainer.targetVolume)` (or `setBrightness` / `setPosition` / seek equivalent) — which is where Kodi's volume was getting overwritten.

`ChargingScreen.qml`'s own `TouchSliderProcessor` Connections (for screensaver speed) was unaffected — both paths share the same broadcast-only hardware signals; the per-feature variants were always *supposed* to bail when the screensaver owned the touchbar, they just weren't bailing *fully*.

### Fix

Added the same one-line guard at the top of `onTouchXChanged` and `onTouchReleased` in all 4 variants — 8 one-liners, identical pattern to the existing `onTouchPressed` guards, zero new drift. No behavior change when the screensaver is inactive.

### Verification

Owner confirmed: "everything seems to work as expected."

## v1.4.8 — Touchbar sensitivity /3 + media-button suppression toggles

### Origin

Two user requests, bundled per `AskUserQuestion` ("Option A").

**Part 1 (sensitivity):** User followup after v1.4.7 — "while the screensaver is running the slider is very sensitive. can we tune it down?" The screensaver speed/density handler at `ChargingScreen.qml:642-666` was mapping raw touchbar pixel delta 1:1 to speed/density units. Over the physical slider's ~90 px usable range (10 → 100 sweep), that felt twitchy.

**Part 2 (media buttons):** User observation that the Kodi integration fork can't selectively strip individual `MediaPlayerFeatures` bits to hide just one of the 4 controls-row icons (shuffle / repeat / media-browser / source-picker). A per-icon feature strip would also break the corresponding command (e.g., stripping `VOLUME_UP_DOWN` to hide the OSD was the same architectural mistake v1.4.1 fixed). UC-side config toggles solve it at the display layer — the same precedent as `Config.showVolumeOverlay` (v1.4.2) and the per-entity `hideVolumeOverlay` (v1.4.4).

### Fix

**Part 1:** Inserted `var scaledDelta = delta / 3;` between the raw delta and the speed/density update in both matrix and starfield branches of `ChargingScreen.qml::onTouchXChanged`. Minimum-movement 3 px dead zone unchanged. Single constant, trivially re-tunable.

**Part 2:** 4 new Q_PROPERTYs on `Config` (`showShuffleButton` / `showRepeatButton` / `showMediaBrowserButton` / `showMediaSourceButton`), QSettings keys `ui/show*Button`, default `true` (preserves upstream behavior on upgrade). Config impl in `config.{h,cpp}` — 4× getter + 4× setter + 4× NOTIFY signal, identical shape to v1.4.2's `showVolumeOverlay` impl. `MediaComponent.qml`:

- Added `import Config 1.0`
- Added `visible: Config.showShuffleButton` to shuffle icon (previously ungated)
- Added `visible: Config.showRepeatButton` to repeat icon (previously ungated)
- AND-chained `Config.showMediaBrowserButton` into the existing browser-icon `visible:` (which already gates on `entityObj.hasFeature(Browse_media) || Search_media`)
- AND-chained `Config.showMediaSourceButton` into the existing source-icon `visible:` (already gates on `hasFeature(Select_source) && sourceList.length !== 0`)

QtQuick.Layouts collapse semantics handle the redistribution — invisible children of a `Layout.fillWidth: true` row don't consume horizontal space, so the remaining visible icons redistribute automatically. No `Layout.preferredWidth: visible ? N : 0` gymnastics needed.

`Ui.qml`: 4 new `ColumnLayout` toggle rows (divider + RowLayout with Switch + helper Text) appended below the `showVolumeOverlay` row, each ~45 lines matching the v1.4.2 template. `volumeOverlaySwitch` extended with `KeyNavigation.down: shuffleButtonSwitch` (previously terminal); chain extends through shuffle → repeat → mediaBrowser → mediaSource (terminal). `Flickable.contentY` clamp bumped 1260 → 1900 for the ~640 px of added content (same 160 px/toggle scaling as v1.4.2's 1100 → 1260).

### Global-master precedence

`Config.showMediaBrowserButton` AND-chains with existing `entityObj.hasFeature(...)` — global master **never unhides** a button the entity chose not to expose. Disabling via Config is strictly subtractive.

Explicitly **not** affected: the press-and-hold gesture on album art that upstream v0.72.0 added (same-purpose entry point for the media browser, still gated on `Browse_media`/`Search_media` features). Called out in the Settings helper text.

### Verification

Part 1: owner tested "works much better. let's keep it like this." Part 2: owner tested all 4 toggles — "the button suppression toggles work."
