## Diagnosis (locked)

Walking the v1.4.9 firmware code:

- `src/core/core.cpp:2100-2163` — `Api::processEntityChange` parses incoming `entity_change` messages. `MsgEventTypes::CHANGE` emits `entityChanged(entity.id, entity)` (line 2150). `MsgEventTypes::NEW` emits `entityAdded(entity)` (line 2142).
- `src/core/core.h:360` — `void entityAdded(Entity entity);` signal declaration.
- **Grep across the entire `src/` tree:** zero connect handlers for `core::Api::entityAdded`. Signal is dead. Inherited from upstream UC fork — the bug pre-exists our v1.4.x work.
- `src/ui/entity/entityController.cpp:429-432` — `onEntityChanged` early-returns silently when `m_entities.contains(entityId) == false`. No log, no fallback.
- Wire trace shows: 15:05:48 integration emits `DELETE` → firmware's `entityDeleted` removes entity from m_entities. 15:07:11 integration emits `NEW` → firmware emits `entityAdded` to no subscriber → m_entities stays empty. 15:07:11+ subsequent `CHANGE` events all hit the silent early-return. The 15:08:01 populated-data-URI `CHANGE` is silently dropped.

Why close+reopen worked: `Activity.qml`'s `includedEntityItem` delegate (lines 660-664) calls `EntityController.load(entityId)` for each included entity → fetches current core state → `addEntityObject` inserts into m_entities. Re-mount runs after the load lands; MediaComponent gets the now-loaded entity, ImageLoader binds, artwork renders. Pre-v1.4.10 `MediaComponent.qml` itself only called `EntityController.get()` (no `load` fallback, no `entityLoaded` signal handler), so it couldn't recover on its own.

## Three-fix bundle

### 1. Connect the orphan `entityAdded` signal (primary, closes the hole)

`src/ui/entity/entityController.cpp` ctor (line 156) — added one connect:

```cpp
QObject::connect(m_core, &core::Api::entityAdded, this, &EntityController::onEntityAdded);
```

New slot delegates to existing `addEntityObject` (which short-circuits if the id is already in m_entities — idempotent on already-loaded entities, safe even if integration emits NEW for an entity it's already pushed):

```cpp
void EntityController::onEntityAdded(core::Entity entity) {
    qCDebug(lcEntityController()) << "Entity added (NEW event):" << entity.id;
    addEntityObject(entity);
}
```

Header declaration mirrors existing slot pattern at `entityController.h:156`. Drift increase: `entityController.{h,cpp}` newly modified upstream files.

### 2. Backstop in `onEntityChanged` (defensive, line 430)

```cpp
if (!m_entities.contains(entityId)) {
    qCWarning(lcEntityController()) << "Late entity_change for unloaded entity, fetching:" << entityId;
    load(entityId);
    return;
}
```

Defends against (a) integrations that emit CHANGE without prior NEW (spec violation, observed in practice), and (b) reconnect races where `m_entities` was just cleared in `onCoreConnected`. Multiple in-flight loads for the same id are safe — `addEntityObject` short-circuits after the first response inserts.

### 3. `MediaComponent.qml` load+`entityLoaded` pattern (defense-in-depth)

`Component.onCompleted` and `onEntityIdChanged` refactored into a shared `ensureEntityLoaded()` helper that calls `EntityController.get(entityId)` first, then `EntityController.load(entityId)` if the get returned null and the id is non-empty. Sibling `Connections { target: EntityController; ignoreUnknownSignals: true; function onEntityLoaded(success, loadedId) }` block re-acquires `entityObj` once the load lands, gated on `loadedId === mediaComponent.entityId`. Pattern matches `SensorWidget.qml:38-46` + `SensorWidget.qml:122-134`. The existing `Connections { target: entityObj }` state-changed scale-animation block preserved verbatim — added the new block as a sibling.
